Flexbox Math: calcSizes
#

The calcSizes function is the heart of our layout system. It solves: given a container and children (some fixed-size, some flexible), how big should each child be?

The Algorithm
# 

1
2
3
4
5
6

pub fn calcSizes(
    children: anytype,
    available: i32,
    gap: i32,
    measureFn: anytype,
) struct { flex_unit: i64, fixed: i32 }

Visual Example
#

Let’s say we have:

  • available = 500px
  • gap = 10px
  • Three children: fixed(100px), flex(1), flex(2)

Step 1: Start with total available space

┌─────────────────────────────────────────────────────────────────┐
│                     available = 500px                           │
└─────────────────────────────────────────────────────────────────┘

Step 2: Subtract gaps between children

With 3 children, we have 2 gaps:

┌─────────────────────────────────────────────────────────────────┐
│                                                     │gap│gap│   │
│              usable space = 480px                   │10 │10 │   │
└─────────────────────────────────────────────────────────────────┘
              ◄──────────────────────────────────────►
                    500 - (10 × 2) = 480

Step 3: Subtract fixed children

The fixed child takes 100px:

┌──────────────────────────────────────────────┬──────────────────┐
│                                              │     fixed=100    │
│           flex_space = 380px                 │    (flex: 0)     │
└──────────────────────────────────────────────┴──────────────────┘
              ◄───────────────────────────────►
                    480 - 100 = 380

Step 4: Divide by total flex units

Total flex = 1 + 2 = 3

flex_unit = 380 ÷ 3 = 126px (rounded down)

Step 5: Final layout

┌────────────┬───┬──────────────────────┬───┬──────────────────────┐
│  flex: 1   │   │       flex: 2        │   │       fixed          │
│  126px     │gap│       252px          │gap│       100px          │
│            │   │   (126 × 2)          │   │                      │
└────────────┴───┴──────────────────────┴───┴──────────────────────┘
     ↑         ↑           ↑              ↑           ↑
  1 unit    10px      2 units          10px     measureFn(child)

var flex_total: i32 = 0;
var fixed: i32 = 0;

for (children) |child| {
    if (child.props.flex > 0) {
        flex_total += child.props.flex;  // Count flex units
    } else {
        fixed += measureFn(child);       // Measure fixed children
    }
}

const gaps: i64 = @as(i64, gap) * @as(i64, @intCast(n - 1));
const avail: i64 = @as(i64, available) - gaps - @as(i64, fixed);
const flex_space: i64 = @max(0, avail);  // Don't go negative
const flex_unit: i64 = if (flex_total > 0) @divFloor(flex_space, flex_total) else 0;

Why i64?
#

The casts to i64 prevent integer overflow. Consider:

  • available = 2000
  • gap = 50
  • n = 100 children
gaps = 50 × 99 = 4950  // Could overflow i32 in edge cases

Rectangle Insets: Padding Made Visual
#

The inset function shrinks a rectangle uniformly from all edges. It’s how we implement padding.

The Function
# 

1

pub fn inset(self: Rect, amount: i32) Rect {
    return .{
        .x = self.x + amount,
        .y = self.y + amount,
        .w = @max(0, self.w - 2 * amount),
        .h = @max(0, self.h - 2 * amount),
    };
}

Why 2× for Width/Height?
#

The inset shrinks from both sides:

                    100px
     ◄──────────────────────────────────►

     ┌──────────────────────────────────┐
     │         │              │         │
     │◄──10px──►◄────80px────►◄──10px──►│
     │  left      remaining     right   │
     └──────────────────────────────────┘

     left side:   -10px
     right side:  -10px
                  ───────
     total:       -20px

If we only subtracted once, the right edge wouldn’t be inset:

     ┌──────────────────────────────────┐
     │         │                        │
     │◄──10px──►◄────────90px──────────►│
     │         │                  ↑     │
     └─────────────────────────── │ ────┘
                                  │
                        touches the edge! (wrong)

DSL Ergonomics: Why tupleToSlice Exists
#

The Problem
#

Zig has two ways to write inline collections:

1
2
3
4
5

// Tuple (anonymous struct) - nice to write
.{ text("A"), text("B"), text("C") }

// Slice - what functions need
&[_]Node{ text("A"), text("B"), text("C") }

Functions need slices to iterate at runtime, but tuples are nicer to type.

The Solution
#

tupleToSlice bridges this gap:

// Without it - ugly explicit arrays:

fn tupleToSlice(tuple: anytype) []const Node {
    const T = @TypeOf(tuple);
    const info = @typeInfo(T);

    // Already a slice? Pass through
    if (info == .pointer) {
        return tuple;
    }

    // It's a tuple - convert at comptime
    if (info == .@"struct" and info.@"struct".is_tuple) {
        const fields = info.@"struct".fields;
        var result: [fields.len]Node = undefined;

        inline for (fields, 0..) |field, i| {
            const val = @field(tuple, field.name);
            // Auto-convert if it has .toNode()
            result[i] = if (@hasDecl(@TypeOf(val), "toNode"))
                val.toNode()
            else
                val;
        }
        return &result;
    }

    @compileError("Expected tuple or slice of Nodes");
}

Visual Data Flow
# 

   .{ text("A"), box(), text("B") }      ← tuple (compile-time struct)
                  │
                  ▼ tupleToSlice

   []const Node{ node1, node2, node3 }   ← slice (runtime iterable)

Event Timing: Hold-to-Repeat
#

The EventManager implements hold-to-repeat behavior, like when you hold a key on your keyboard.

The Timeline
# 

User holds mouse button on "+" button:

  Click!                     repeat  repeat  repeat  repeat
    │                          │       │       │       │
    ▼                          ▼       ▼       ▼       ▼
────●────────────────────────────●───────●───────●───────●────▶ time
    │                          │
    │◄────── 0.35s delay ─────►│◄─ 0.06s ─►│
    │        (hold_delay)           (hold_rate)

The State Machine
# 

┌─────────────────────────────────────────────────────────────┐
│                      EventManager                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   handlers: [ increment, decrement, submit, ... ]           │
│                  ▲                                          │
│                  │ index                                    │
│   event_queue: [ 0, 0, 1, 0 ]   ← "call handler 0,0,1,0"   │
│                                                             │
│   held_handler: 0               ← currently held button     │
│   hold_time: 0.42               ← how long held             │
│                                                             │
└─────────────────────────────────────────────────────────────┘

The Repeat Logic
#

The tricky part is firing at regular intervals:

1
2
3
4
5
6
7
8

const t = self.hold_time - hold_delay;      // Time since delay ended
const prev_t = t - dt;                       // Previous frame's time
const curr = @intFromFloat(t / hold_rate);   // Current interval
const prev = @intFromFloat(prev_t / hold_rate); // Previous interval

if (curr > prev) {
    self.queue(handler_id);  // Crossed a boundary - fire!
}

            │      │      │      │      │
curr:       0      1      2      3      4
prev:      -1      0      1      2      3
            │      │      │      │      │
fire?      ✓      ✓      ✓      ✓      ✓

Fires when crossing each 0.06s boundary

Why Queue Instead of Direct Call?
# 

❌ Direct call - unpredictable order:
   button1.onClick() → modifies state
   button2.onClick() → reads stale state?

✓ Queue then flush - predictable:
   queue(0)          ─┐
   queue(1)           │ collect all
   queue(0)          ─┘
   flush()           → process in order, all at once

Row vs Column
#

The only difference is which axis they work on:

renderCol                          renderRow
    │                                  │
    ▼                                  ▼
┌────────┐                     ┌────┬────┬────┐
│ child1 │                     │ c1 │ c2 │ c3 │
├────────┤  vertical           └────┴────┴────┘
│ child2 │  stacking              horizontal
├────────┤                        stacking
│ child3 │
└────────┘

calcSizes uses content.h        calcSizes uses content.w
measureH for fixed sizes        measureW for fixed sizes
increments y                    increments x
